# 超大規模音声ライブラリ対応アップデート計画 (2026-03-15)

## 0. 目的（必達）

本アプリは、以下を同時に満たすことを正式目標とする。

- 対象形式: `wav / mp3 / m4a / ogg`
- **大量**: リストに **30万ファイル**が存在しても実用的に操作できること
- **高速**: 起動/フォルダ読み込み直後でも、**3秒以内に作業開始**できること（全量デコード完了を待たない）
- **長時間安定**: 長時間運用（連続編集・連続再生・大量スクロール）でも破綻しないこと

さらに、音響品質に関して次を**絶対条件**とする。

- 再生時に **音程・再生速度・サンプルレート・チャンネル構成が意図せず変化しない**
- まだ完全ロードが終わっていない状態でも、再生経路での暗黙変換により音が変わらない
- 編集操作をしていない限り、入力音声と同一内容が再生される（ノンデストラクティブ原則）

---

## 1. 非機能要件（SLO/品質ゲート）

## 1.1 性能SLO

- リスト初期表示（10万〜30万件）
  - UIフリーズなし
  - 初回スクロール可能まで <= 3秒
- 単一ファイル選択後のエディタ作業開始
  - 低解像度波形/代理表示で <= 3秒
  - 高解像度データは段階的に置換
- メモリ上限管理
  - 長時間利用時にメモリが単調増加し続けない（キャッシュ上限 + エビクション）

## 1.2 音響同一性SLO（最重要）

- 非編集再生で以下が常に一致
  - 再生サンプルレート（入力メタ情報）
  - ピッチ（検証許容誤差内）
  - 再生時間（速度変化なし）
- 未ロード区間を跨ぐ再生でも、ロード完了前後で音質ジャンプや速度変化を起こさない
- デコード経路差（先行プレビュー経路 / 本ロード経路）で結果が一致すること

## 1.3 安定性SLO

- 8時間以上の連続運用でクラッシュゼロ
- 大量ジョブキャンセル/再投入でもデッドロックゼロ
- バックグラウンド処理が失敗してもUI操作継続可能

---

## 2. 基本方針

1. **UIスレッドをブロックしない**
   - 重いI/O、デコード、波形要約、外部データ処理は必ずバックグラウンド化
2. **可視範囲優先 + 段階的表示**
   - まず作業可能状態を作り、精度は後段で引き上げる
3. **音の不変性を最優先**
   - 最適化より「音が変わらない」ことを優先
4. **検証駆動**
   - 各フェーズで性能計測 + 音響同一性テストを必須化

---

## 3. フェーズ計画

## Phase 0: 計測基盤と品質ゲートの先行整備

### 目的
- 最適化前に、現状性能と音響同一性を定量化する。

### 実装
- フレーム時間計測（P50/P95/P99）と処理内訳ログを追加
- リスト件数別（1万/10万/30万）のシナリオベンチを固定化
- 音響同一性テスト基盤を追加
  - 再生経路の出力サンプル比較（参照デコードと一致検証）
  - SR/テンポ/ピッチの回帰テスト

### 完了条件
- CIまたはローカル定型コマンドで性能・音響品質の回帰検出が可能

## Phase 1: 30万件リスト対応（仮想化と索引）

### 目的
- 大量ファイルでもリスト操作が詰まらない状態を作る。

### 実装
- リスト描画の完全仮想化（可視行のみ構築）
- フィルタ/ソート用の軽量索引構造を導入（文字列再生成を抑制）
- サムネ/メタ読み込みは遅延実行（表示要求された分だけ）
- バックグラウンド走査ジョブを分割し、進捗・キャンセル対応

### 完了条件
- 30万件データでスクロール/選択/検索が実用速度

## Phase 2: 3秒以内作業開始（段階的ロード）

### 目的
- 全量ロード前でもエディタ操作を開始できるようにする。

### 実装
- 波形の段階的解像度（粗→中→精）を採用
- 先行プレビュー用要約データ（min/max等）を優先生成
- 本デコード完了後にシームレス置換（カーソル位置・選択範囲を維持）
- 長尺音声はチャンク単位で逐次ロード

### 完了条件
- 任意ファイル選択から3秒以内に波形上で基本操作可能

## Phase 3: 再生経路の音響不変性ロックダウン（最重要）

### 目的
- 未ロード時を含め、再生音が絶対に変化しない土台を確立する。

### 実装
- 再生パイプラインを「同一マスタークロック + 同一SR方針」に統一
- 暗黙リサンプルの発生箇所を全列挙し、許可制へ変更
- デコーダ差分吸収レイヤを導入（形式差を正規化しつつ音を変えない）
- 「未ロード区間の代替データ再生」を禁止し、必要なら待機/無音を明示制御
- ピッチ/速度/SR変更機能は通常再生経路から明確分離

### テスト（必須）
- 形式別（wav/mp3/m4a/ogg）同一素材でAB比較
- ロード中→ロード完了跨ぎ再生の連続性検証
- 長時間連続再生でドリフト検証

### 完了条件
- 非編集再生で音程・速度・SRの回帰ゼロ

## Phase 4: キャッシュ基盤（長時間安定 + 再起動高速化）

### 目的
- 同一ファイル再訪時の待ち時間を削減し、長時間運用を安定化する。

### 実装
- 波形要約の永続キャッシュ（multi-resolution）
- キャッシュ整合性（version/checksum）
- 原子的更新（temp -> sync -> rename）
- メモリキャッシュとディスクキャッシュの2段管理
- LRU/サイズ上限によるエビクション

### 完了条件
- 再起動後の再表示が大幅短縮
- 長時間運用でもメモリ/ディスク使用量が制御下

## Phase 5: SIMD・並列最適化（ホットスポット限定）

### 目的
- 本当に重い区間だけを安全に高速化する。

### 実装
- まず自動ベクトル化が効く形へループ整理
- 必要箇所に限定してSIMD導入（安全なfeature検出 + フォールバック）
- 波形要約計算をデータ並列化
- UI反映はメッセージ経由に限定し、ロック競合を最小化

### ガードレール
- SIMD/並列化で音響同一性を壊さない（丸め差許容範囲を事前定義）
- 配布互換性を損なう固定ISAビルドを避ける

### 完了条件
- P95/P99フレーム時間の有意改善
- 音響同一性テストを全通過

## Phase 6: 非同期I/O最適化とジョブ制御

### 目的
- 体感待ち時間を減らし、操作中断時のレスポンスを高める。

### 実装
- I/O待ちを非同期化（ファイル列挙・メタ取得・読み込み）
- CPU重処理とI/O待ち処理の実行基盤を分離
- ジョブ優先度（可視範囲 > 次に見る候補 > それ以外）
- キャンセル伝播を徹底し、古いジョブ結果の誤反映を防止

### 完了条件
- 連続操作時の「待ち戻り」削減
- キャンセル後の復帰が即時

## Phase 7: 長時間運用・障害耐性強化

### 目的
- 実使用での破綻ポイント（リーク、デッドロック、破損）を潰す。

### 実装
- soak test（8〜24時間）と大量操作リプレイ
- 異常系（壊れたファイル、途中切断、権限なし）の回復手順整備
- セッション復元の堅牢化（クラッシュ後の再開）
- ログ/診断情報の整備（再現用最小パッケージ出力）

### 完了条件
- 長時間試験の安定性基準を満たす

## Phase 8: 仕上げ（UX・リリース運用）

### 目的
- 性能と安全性を維持したまま、現場で使える品質に仕上げる。

### 実装
- 進捗表示・キャンセルUI・失敗時導線の改善
- チューニング項目のプリセット化（安全設定を既定）
- 互換性テスト（OS/CPU/ファイル形式）
- リリースチェックリスト整備

### 完了条件
- ユーザーが「大量・高速・長時間」を破綻なく再現できる

---

## 4. 音を絶対に変えないための実装規約（必読）

- 通常再生パスで、以下の暗黙処理を禁止
  - 自動ピッチ補正
  - 自動タイムストレッチ
  - 意図しないリサンプル
- 編集機能（ピッチ/速度/SR変換）は、明示操作時のみ有効化
- 「ロード中だから代替アルゴリズム」は許容するが、**出力音が変わらない同等処理**のみ許可
- デコード器切替時は、回帰テスト（波形差分・長さ・基本周波数）を必須化
- 不一致検出時は最適化をロールバックし、品質優先でリリース判断

---

## 5. 優先順位（実装順）

1. Phase 0（計測・音響テスト）
2. Phase 3（音響不変性ロックダウン）
3. Phase 1（30万件リスト）
4. Phase 2（3秒以内作業開始）
5. Phase 4（キャッシュ）
6. Phase 5（SIMD/並列）
7. Phase 6（非同期I/O）
8. Phase 7〜8（耐性・仕上げ）

※理由: 速さは重要だが、再生音が変わる実装は採用しないため、音響不変性の土台を先に固定する。

## 5.1 直近実装メモ（2026-03-15）

- `src/audio_io.rs` のプロキシ波形プレビューで、proxy側の `sample_rate` を推定値へ縮退させず、常に元音声の `source_sample_rate` と同値で返すよう変更。
- これにより、部分ロード用プレビューデータが再生経路に混入した場合でも、サンプルレート解釈の不一致で音程/速度が変わるリスクを下げる。
- 併せてユニットテストを追加し、「source SR固定」「0回避」を検証。
- Progressive decode（m4a / symphonia / multi-progressive）のチャンク生成中に、サンプルレートやチャンネル数が途中で変化した場合は即時エラーにするガードを追加（ロード中でも再生条件が揺れないようにする）。加えて、通常の `decode_m4a_fdk` 経路でも同様のドリフト検出を追加。
- Linux デバッグ向けに `glow/wgpu` feature へ `eframe/x11` と `eframe/wayland` を含め、`winit` のプラットフォーム未選択エラーを避ける設定を追加。
- `decode_audio_multi_progressive` の m4a 分岐を段階チャンク処理へ寄せ、`prefix_secs` / `emit_every_secs` / cancel を有効化（全量デコード完了待ちを避けて作業開始を早める）。

## 6. マイルストーン案

- M1: 計測基盤 + 音響同一性テスト導入
- M2: 30万件リストで実用操作達成
- M3: 3秒以内の編集開始達成
- M4: ロード中/完了跨ぎで音響変化ゼロ
- M5: キャッシュ導入で再表示高速化
- M6: P95/P99改善 + 長時間安定性クリア
